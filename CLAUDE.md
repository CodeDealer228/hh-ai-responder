# hh-ai-responder

CLI-инструмент для автоматических откликов на hh.ru с помощью AI. Форк
[s3rgeym/hh-ai-responder](https://github.com/s3rgeym/hh-ai-responder) (upstream remote
`upstream`), рабочая копия — `origin` = `https://github.com/CodeDealer228/hh-ai-responder`.

Go 1.25, вся логика в одном файле [main.go](main.go) (~3300 строк), **только стандартная
библиотека**, зависимостей и `go.sum` нет.

## Архитектура за 30 секунд

- Авторизация — **только через `cookies.txt`** (Netscape-формат, экспорт браузерным
  расширением "Get cookies.txt LOCALLY" после ручного логина на hh.ru). Встроенного
  OAuth/login-флоу нет.
- `HHAIResponder` (main.go:1336) — главный объект, держит cookie jar, HTTP-клиент,
  распарсенные данные резюме, AI-клиент.
- `Run()` крутит 4 независимых горутины с разным интервалом: тач резюме (4ч), статус
  поиска работы (24ч), отклики (12ч), чат (15м).
- `MemoryPersistentJar` (main.go:2806) — самописный cookie jar поверх `cookies.txt`
  вместо стандартного `net/http/cookiejar`. Домен-мэтчинг в `Cookies()` (main.go:2904)
  завязан на конкретный host запроса.
- AI — любой OpenAI-совместимый `/v1/chat/completions` (Ollama, OpenAI, OpenRouter и
  т.д.), настраивается через `HH_AI_BASE_URL`/`HH_AI_MODEL`/`HH_AI_API_KEY`.

## Конфигурация

Флаги и переменные окружения — см. `parseConfig()` (main.go, поиск по `flag.StringVar`)
и [example.env](example.env)/[README.md](README.md). Рабочий `.env` и `cookies.txt` в
`.gitignore`, не коммитятся.

Важные флаги/переменные:

| Переменная | Флаг | Назначение |
|---|---|---|
| `HH_SEARCH_URL` | `-u` | URL поиска вакансий (`hh.ru/search/vacancy?...`) |
| `HH_RESUME` | `-r` | ID/хэш резюме; пусто = последнее обновлённое |
| `HH_AI_BASE_URL/MODEL/API_KEY` | `-ai-*` | AI backend |
| `HH_CONTACTS` | `-contacts` | Контакты, которые бот может передать работодателю |

`docker-compose.yml` запускает `-o results.json -l debug -force-letter` в режиме
`restart: unless-stopped` — то есть бесконечный цикл с реальными действиями на реальном
аккаунте. **Флага dry-run и лимита на число откликов за прогон нет** — `-mr` это
фильтр по количеству чужих откликов на вакансию, а не лимит своих действий.

### Контакты работодателя — не реализовано

В коде нет ни одной строчки, обращающейся к БД. `AutoRespondChats` знает
`ContactName`/`CompanyName`/текст сообщений (`GetChatData`/`JoinChatMessages`,
main.go:1144-1280), но эти данные никуда не пишутся, кроме JSON-лога событий
(`results.json`, тоже в `.gitignore`). Если делать — сюда: после получения текста
сообщения работодателя доставать телефон/email/telegram (regex, по образцу
`latesteResumeHashRegexp`/`userIdRegexp` в начале файла) и апсертить в БД по `chat_id`.
Для БД без CGO (Dockerfile собирает с `CGO_ENABLED=0`) нужен чистый Go-драйвер, например
`modernc.org/sqlite`, а не `mattn/go-sqlite3`.

## Баги, найденные и исправленные в этой сессии

1. **Резюме не читалось для аккаунтов с одним резюме.** hh.ru сейчас редиректит
   `/applicant/resumes` → `/applicant/profile/me`, а на этой странице список резюме не
   встроен в JSON (`{"redirectConfig":...}`), который парсит `LoadProfileData`
   (main.go:2025). Без резюме `NewHHAIResponder` падает с `resume not found` — это
   ломало вообще любой запуск для таких аккаунтов, включая дефолтный `example.env`.
   Фикс: `FetchResumeSummary` (main.go:2371) — фолбэк, читает конкретное резюме
   напрямую с `/resume/<hash>` (тот же паттерн, что уже использовал
   `GetResumeExperience`). **Требует явно заданного `HH_RESUME`** — без него нечего
   фетчить, авто-определение "последнего" резюме на такой странице тоже не работает.
2. **Домен-мэтчинг кук.** `MemoryPersistentJar.Cookies()` (main.go:2904) не подставляет
   куки с доменом `.hh.ru` (с точкой — так реально экспортируются `hhtoken`/`hhuid`) в
   запросы к голому `hh.ru` (без поддомена) — только к поддоменам. Обойдено на уровне
   конфига: `HH_SEARCH_URL` указывает на региональный поддомен (`spb.hh.ru` и т.п.),
   не на голый `hh.ru`. Сам баг в коде не тронут (не было уверенности, что фикс не
   сломает другой сценарий использования домен-мэтчинга) — если увидите `403` на
   `/applicant/resumes` с валидными свежими куками, проверьте это первым делом.
3. **Перепутаны местами `salary`/`experience`** при вызове `GenerateLetter` в
   `ApplyVacancies` — из-за этого в письмо под "Зарплата:" попадал весь текст опыта
   работы, а "Твой опыт:" оставался пустым. Исправлено.
4. **Не было `%%`-экранирования в промпте письма** — при добавлении few-shot-примера с
   текстом "40%" сломало позиционные аргументы `fmt.Sprintf` (поля съезжали друг в
   друга, в лог попадало `%!s(MISSING)`). Промпт переписан на конкатенацию строк вместо
   одного большого `Sprintf`, так что сырой текст в примерах/резюме/описании вакансии
   больше не может сломать форматирование.
5. **Резюме читалось один раз при старте** и кэшировалось в памяти на весь процесс
   (который может жить неделями под `restart: unless-stopped`) — правки резюме на
   сайте не подхватывались. Добавлен `RefreshResumeData` (main.go:1656), вызывается в
   начале каждого цикла откликов (каждые 12ч).

## Известные ограничения (не баги кода, но важно знать)

- **AI backend может галлюцинировать по-разному в зависимости от модели.** С
  `openrouter/free` (авто-роутер бесплатных моделей OpenRouter) вживую поймал два
  разных сбоя:
  - модель иногда возвращает служебный модерационный вердикт вместо реального текста
    (`"User Safety: unsafe\nSafety Categories: PII/Privacy"`) — это уходило
    работодателю как обычное сообщение. Исправлено: `moderationArtifactRegexp`
    (main.go, рядом с `latesteResumeHashRegexp`) отбраковывает такие ответы, для
    чата — фолбэк на случайную строку из [questions.txt](questions.txt).
  - "thinking"-модели иногда пишут в ответ черновик рассуждений вместо готового текста
    письма (`"We need to produce a cover letter..."`, `"Sentence 1: ..."`). Исправлено:
    `letterArtifactRegexp` в `GenerateLetter` отбраковывает такие ответы (вакансия
    пропускается, как при обычной ошибке AI).
  - hh-тестирование: тот же `openrouter/free` иногда возвращает невалидный JSON
    (массив без `[...]`) — заявка в этом случае корректно не отправляется, ошибка
    логируется, приложение не падает.
- **OpenRouter free tier: 50 запросов/день** (или 1000/день после разовой покупки
  $10 кредитов), 20 RPM. При `-force-letter` это письмо на КАЖДЫЙ отклик — на активном
  поиске лимит выжирается быстро. Модель `openrouter/free` — авто-роутер по всем
  доступным бесплатным моделям, что даёт больше шансов не упереться в лимит одной
  конкретной модели, но не спасает от общего дневного лимита аккаунта OpenRouter.
- Промпт чата прямо инструктирует AI соглашаться на все условия и утверждать наличие
  любых навыков — осознанный выбор апстрима, не баг.
- `questions.txt` — запасные сообщения для чата на случай мусорного/пустого ответа AI,
  по одному на строку, выбор случайный. Не факт, что случайный вопрос будет уместен как
  ответ на конкретное сообщение работодателя — это осознанный компромисс ради того,
  чтобы чат не замолкал совсем, а не смысловая замена AI-ответу.

## Локальный запуск

```sh
docker compose up -d --build     # поднять как сервис, см. docker-compose.yml
docker compose logs -f           # -l debug уже включен в command
docker compose stop              # остановить, не удаляя контейнер
```

`cookies.txt` и `.env` кладутся в корень репозитория (volume `.:/app`). При `403` на
`/applicant/resumes` с валидными куками — см. баг №2 выше, попробуйте региональный
поддомен вместо `hh.ru` в `HH_SEARCH_URL`.
